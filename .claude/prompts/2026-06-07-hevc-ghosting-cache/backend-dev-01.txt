## TÂCHE : Analyser les buffers persistants dans DataChannelRelay et StreamRelay

BUG CRITIQUE : Un stream HEVC WebRTC montre la dernière frame du stream d'HIER en première image. C'est un cache persistant qui traverse les sessions.

### Ce que tu dois faire

1. **Lis COMPLÈTEMENT** les fichiers suivants :
   - `d:\Code\moonlight-web-deepseek\backend\src\streaming\DataChannelRelay.cpp`
   - `d:\Code\moonlight-web-deepseek\backend\src\streaming\DataChannelRelay.h`
   - `d:\Code\moonlight-web-deepseek\backend\src\streaming\StreamRelay.cpp`
   - `d:\Code\moonlight-web-deepseek\backend\src\streaming\StreamRelay.h`

2. Pour chaque fichier, identifie TOUS les buffers membres qui pourraient persister entre deux streams :
   - QByteArray, std::vector<uint8_t>, std::array, char*, uint8_t*, ou tout buffer alloué dynamiquement
   - Variables membres qui stockent des données de frame
   - Tout buffer réutilisé (pas de clear/reset au start)

3. Vérifie le cycle de vie :
   - L'objet DataChannelRelay est-il DÉTRUIT et RECRÉÉ entre deux streams, ou RÉUTILISÉ ?
   - Si réutilisé, est-ce que tous les buffers sont vidés au start() ?

4. Cherche SPECIFIQUEMENT :
   - Tout buffer qui accumule des NAL unit fragments
   - Tout buffer de trame incomplète en attente
   - Tout cache de frame pour réorganisation (reorder)

5. **Cherche aussi dans `backend/src/streaming/Session.cpp`** — le cycle de vie complet

Un listing de TOUS les buffers potentiellement problématiques, avec :
- Le fichier, la ligne, la variable
- S'il est vidé ou non au start d'un nouveau stream
- S'il peut expliquer une frame d'hier qui apparaît aujourd'hui
