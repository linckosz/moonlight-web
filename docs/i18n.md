# Internationalisation (i18n) — MoonlightWeb

Le frontend est entièrement traduisible. L'anglais (`en`) est la langue de
référence ; le français (`fr`) est fourni. Tout est piloté par des fichiers de
**variables de langue** (`frontend/locales/*.json`) et un petit runtime maison,
avec **Tolgee self-hosted** comme interface web de gestion des traductions.

---

## 1. Architecture runtime

Pas de build step ni de bundler : modules ES6 servis statiquement. Le runtime est
donc volontairement léger (≈ 90 lignes), sans dépendance externe.

| Élément | Rôle |
|---|---|
| `frontend/js/i18n/i18n.js` | Le moteur : `init`, `t`, `applyDOM`, `setLanguage`, `getLanguage`, `AVAILABLE_LANGUAGES`. |
| `frontend/locales/en.json` | **Catalogue de référence** (toutes les clés existent ici). |
| `frontend/locales/fr.json` | Traduction française (même arborescence de clés). |

### Boot

`frontend/js/app.js` charge les traductions **avant** le premier rendu :

```js
await i18nInit();   // fetch locales/<lang>.json (+ en.json en fallback)
applyDOM(document); // traduit le HTML statique de index.html
MoonlightApp.init();
```

### Utiliser une traduction

- **Dans le JS** (templates des vues, toasts) :
  ```js
  import { t } from '../i18n/i18n.js';
  t('hosts.title');                       // "Hosts" / "Hôtes"
  t('launch.launching', { name: app.name }); // interpolation {{name}}
  ```
- **Dans le HTML statique** (`index.html`) :
  ```html
  <span data-i18n="footer.tagline">…</span>
  <button data-i18n-attr="title:header.serverSettings">…</button>
  ```

`t(key, vars)` résout une clé pointée (`"a.b.c"`) dans la langue active, avec
interpolation `{{var}}`. Clé absente → fallback anglais → clé brute (l'UI ne
casse jamais).

### Choix et persistance de la langue

Ordre de détection : `localStorage["mw-lang"]` → `navigator.language` → `en`.
Le sélecteur est dans **Paramètres de streaming** (section *Langue*) ;
`setLanguage(code)` enregistre le choix et recharge la page.

### Ajouter une langue

1. Ajouter une entrée dans `AVAILABLE_LANGUAGES` (`i18n.js`) :
   `{ code: 'es', label: 'Español' }`.
2. Créer `frontend/locales/es.json` (copie de `en.json`, traduite).

Rien d'autre à câbler. Vérifier avec le script ci-dessous.

### Contrôle d'intégrité

```bash
cd frontend && node scripts/check-i18n.cjs
```

Vérifie : JSON valides · **parité des clés** entre toutes les locales · chaque
`t('clé')` du code a bien une entrée dans `en.json`. Code de sortie 1 si problème
(utilisable en pre-commit / CI).

## 2. Gestion via Tolgee (interface web)

Le runtime ne dépend **pas** de Tolgee : Tolgee est un éditeur découplé qui se
synchronise avec `frontend/locales/*.json` via la **Tolgee CLI**. Le format
utilisé est `JSON_I18NEXT` (clés imbriquées + interpolation `{{var}}`), identique
à nos fichiers — aucune conversion.

### Lancer Tolgee

```bash
cd tolgee && docker compose up -d
# Interface : http://localhost:8085   (admin / admin — à changer)
```

### Configuration CLI

`frontend/.tolgeerc.json` décrit le projet, le format et les fichiers à
pousser/tirer. Renseigner `projectId` (créé dans l'UI) et un token API
(`Settings → API keys`) :

```bash
export TOLGEE_API_KEY=tgpat_xxx   # ou: tolgee login
cd frontend
```

### Workflow

1. **Import initial** (envoyer le catalogue existant dans Tolgee) :
   ```bash
   npx @tolgee/cli push
   ```
2. **Éditer / traduire** dans l'interface web (http://localhost:8085) — ajout de
   clés, traductions, nouvelles langues, relecture.
3. **Récupérer** les traductions vers le repo :
   ```bash
   npx @tolgee/cli pull
   node scripts/check-i18n.cjs   # parité + clés
   ```
4. Bump du `?v=` dans `index.html` si besoin de forcer le rechargement client.

> Une nouvelle langue ajoutée côté Tolgee arrive comme `locales/<code>.json` au
> `pull`. Penser à l'ajouter aussi dans `AVAILABLE_LANGUAGES` pour qu'elle
> apparaisse dans le sélecteur.

## 3. Namespaces du catalogue

`common`, `header`, `footer`, `appError`, `hosts`, `apps`, `login`, `pairing`,
`settings`, `admin`, `stream`, `transport`, `launch` (279 clés).

Les chaînes purement techniques et identiques dans toutes les langues (noms de
protocoles `WebRTC DataChannel (UDP)`, glyphes `⛶`/`⌨`, unités `Mbps`/`FPS`,
codecs) sont gérées au cas par cas : protocoles laissés littéraux, glyphes
conservés, unités interpolées.
